# Component consistency

Рабочий принцип. Применяется ко всем визуальным элементам, которые появляются
на странице больше одного раза.

## Правило

**Если приём утверждён — он живёт в одном компоненте, и все тайлы используют
этот компонент. Если приём меняется — меняется везде.**

Никаких «копий с небольшими отличиями», «локальных версий для этого тайла»,
«почти такого же, но с процентом». Одна форма → один компонент → один импорт.

## Почему

1. **Пользователь привыкает к одной форме.** Дельта в `+8` в Brands и дельта
   в `+12%` в Price — это один и тот же визуальный объект (цвет, высота,
   padding, line-height). Разный контент, одинаковая плашка. Повторение формы
   → узнаваемость за 100ms → отчёт читается быстрее.

2. **Баги ловятся в одном месте.** Когда дельта визуально сидит ниже центра,
   мы правим `DeltaChip` в `common.tsx` — и это чинится во всех 15 тайлах
   сразу. В старом мире было 6 локальных копий `DeltaChipPct` с `py-0.5`
   и одна правильная `DeltaChip` с `py-1 leading-none`. Отсюда — «в одной
   плитке норма, в другой — дельта проваливается». Неприемлемо на живом
   дашборде.

3. **Код-ревью не краснеет.** Если 15 плиток используют 15 локальных копий
   одного паттерна — значит, «мы вайб-кодили», говорит senior. Если все 15
   импортируют из `common.tsx` — значит, у нас есть design system.

## Как это выглядит в проекте

### Shared components → `prototype/src/components/common.tsx`

Единый источник правды для визуальных примитивов:

- `DeltaChip({ delta })` — абсолютная дельта (`+8`, `−3`, `±0`).
- `DeltaChipPct({ pct })` — процентная дельта (`+17%`, `−28%`).
- `BrandLogo({ name, size })` — лого бренда с fallback на `BrandMonogram`.
- `BrandMonogram({ name, size })` — инициалы на gold-круге.
- `WatchPlaceholder({ name, size })` — плейсхолдер часов.
- `Eyebrow`, `SectionTitle` — типографические примитивы.

Любой тайл, которому нужна дельта / лого / eyebrow — импортирует отсюда,
не пишет свой.

### Approved patterns → `docs/principles/`

Паттерн верстки, за который мы проголосовали, живёт здесь отдельным md-файлом:

- `list-tile-row-pattern.md` — per-row grid для list-тайлов (Brands,
  Collections, Dials, Straps, Materials, Functions, Heights, Movement legends).

Когда в проекте появится ещё один повторяющийся паттерн (column chart,
histogram, hero-block) — он пишется сюда, и все реализации обязаны ему
следовать.

## Когда разрешено отступить

Редко. Три легитимные причины:

1. **Sandbox-эксперимент.** Файл с суффиксом в названии (`BrandsTileE`,
   `DialsTileSandbox`) может пробовать новый паттерн — на то он и sandbox.
   Но если эксперимент приняли — он либо заменяет канон (обновляем принципы
   + мигрируем все плитки), либо откатывается и живёт с `archived: true` в
   `tileRegistry`.

2. **Радикально другая семантика.** Например, `SectionTitle` с `count` в
   углу — не `Eyebrow`, у него другая роль. Общий компонент не подходит,
   делаем новый общий компонент с другим именем.

3. **Временный обход для проверки гипотезы.** Срок жизни — один рабочий
   день. Помечаем `// TEMP: будет слито в common после ревью`. Если через
   день не слили — это долг, и он горит.

## Антипаттерны

### ❌ «Локальная копия с минимальным отличием»

```tsx
// DiameterTileB.tsx
function DeltaChipPct({ pct }) {
  // ...same code as 5 other files, slightly different padding...
}
```

Откуда берётся: «мне нужен процент, а в common.tsx абсолютное число — не
хочу ломать Brands». Правильный ход: добавить `DeltaChipPct` в common,
не плодить копии.

### ❌ «Почти такой же, но с justify-self-end»

```tsx
// BrandsTileD.tsx
<li className="contents">
  <span className="num justify-self-end">{count}</span>
</li>

// CollectionsTileB.tsx
<li className="grid grid-cols-[...] items-center">
  <span className="num">{count}</span>
</li>
```

Две «почти одинаковые» плитки, на инспекции подсветка отступов ведёт себя
по-разному, дельты выравниваются неконсистентно. Правильный ход: выбрать
один паттерн (per-row grid выиграл — см. `list-tile-row-pattern.md`) и
мигрировать всё туда.

### ❌ «Это только для этой плитки, остальные потом»

Чаще всего «остальные потом» не наступают. Либо сразу делаем везде, либо
не делаем совсем. Если на «везде» нет времени — значит, изменение
слишком большое и его надо разбить.

## Процесс

### Когда добавляешь новый визуальный элемент

1. Проверь `common.tsx` — возможно, уже есть.
2. Если нет — подумай, встретится ли он ещё где-то. Если да (и почти
   всегда — да) — сразу пиши в `common.tsx`, импортируй из тайла.
3. Если паттерн сложнее одного компонента (верстка строки, layout сетка) —
   заводи файл в `docs/principles/` одновременно с кодом.

### Когда меняешь утверждённый приём

1. Обнови компонент в `common.tsx` или принцип в `docs/principles/`.
2. Пройди все места использования — убедись, что ничего не развалилось.
3. Если принцип менялся формально (например, «теперь count сидит в
   отдельной колонке, не в name-stack») — обнови `docs/principles/` и
   мигрируй все тайлы в одном коммите.

### Когда находишь неконсистентность

1. Не молчи — пофикси.
2. Правку делай в источнике (`common.tsx` или principle doc), не в
   потребителе.
3. Если нашёл дубликат — слей в общий компонент, удали локальные копии.

## Contributor checklist

Перед каждым PR/коммитом, который меняет визуальный элемент:

- [ ] Компонент импортируется из `common.tsx`, а не скопирован локально?
- [ ] Если ввёл новый паттерн — добавил md в `docs/principles/`?
- [ ] Если поменял утверждённый приём — прошёл все места использования?
- [ ] В списках-тайлах применён per-row grid паттерн
  (см. `list-tile-row-pattern.md`)?
- [ ] Дельты рендерятся через `<DeltaChip>` / `<DeltaChipPct>` из common,
  без локальных копий?
